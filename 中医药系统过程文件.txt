中医药辅助诊疗系统 · 项目需求与过程摘要
==============================================

一、项目概述
-----------
基于 31 万条真实中医医案数据（300MB，16 个 JSON 分片），构建中医药辅助诊疗 Web 应用。
技术栈：Python Flask 后端 + 原生 HTML/CSS/JS 前端 + DeepSeek API（LLM）+ Render 云部署。
GitHub 仓库：https://github.com/MaYan413/tcm-search

二、三版本架构（最终确定）
-------------------------
1. 专业版 (/)：31万医案检索 + 节气养生 + 九种体质测评，无 AI 聊天，无跨版本按钮
2. 大众版 (/llm 或 /dazhong)：纯 DeepSeek AI 诊疗对话，无跨版本按钮，服务端统一管理 Key
3. 混合版 (/hunhe + /hunhe-pro)：大众版界面 + 右上角「专业版」按钮可切换到专业版检索界面，双向互跳

三、关键技术决策
---------------
1. API Key 管理：服务端通过环境变量 LLM_API_KEY 统一管理，用户端无需输入
2. 内存优化（核心难点）：Render 免费版仅 512MB，300MB 数据不能全量加载
   - 最终方案：纯文本扫描定位 → str.find() 找到匹配位置 → 只 json.loads() 那一条记录
   - 逐文件处理 + gc.collect() 即时释放，内存稳定在 ~30MB
3. 部署：GitHub Push → Render 自动检测 → 自动部署（render.yaml 配置）
4. 路由设计：/api/search（文本扫描搜索）、/api/formula（定位单条解析）、/api/chat（DeepSeek API 代理）
5. UI 主题：绿色新中式风格（#1a5a2a/#2d7a3a），LLM 回复自动去除 ** 星号、【标题】自动加粗加大

四、文件结构
-----------
- app.py：Flask 后端（路由、搜索、AI 代理、静态文件服务）
- index.html：专业版（31万医案检索前端）
- llm.html：大众版（AI 诊疗前端）
- 混合版.html：混合版（大众版界面 + 专业版跳转按钮）
- 混合版专业.html：混合版的检索界面（专业版界面 + 大众版跳转按钮）
- data/*.json：16 个分片医案数据文件
- render.yaml：Render 部署配置
- requirements.txt：Python 依赖（flask, flask-cors, openai）

五、LLM 版 Prompt 设计要求
-------------------------
- System Prompt 定位于中医诊疗 AI
- 回复中严格禁止出现 * 星号
- 小标题使用【标题】中文方括号格式
- 前端自动清洗：移除所有 ** 标记，将【标题】渲染为加粗加大字号
- 注意：HTML 转义必须先于 span 标签注入执行，避免 <span> 标签泄漏

六、混和版导航要求
------------------
- /hunhe（大众版界面）：右上角仅有「📋 专业版」→ 跳转 /hunhe-pro
- /hunhe-pro（专业版界面）：右上角仅有「大众版」→ 跳转 /hunhe
- 单独的专业版(/)和大众版(/llm)无任何跨版本按钮

七、已知坑点
-----------
1. GitHub 国内网络不稳定，push 需多次重试
2. Render 免费版 15 分钟无访问自动休眠，首次打开需等 30-50 秒唤醒
3. 中文文件名在 Linux 环境可能编码异常，建议用英文/拼音文件名
4. E 盘和 D 盘各有一份代码，实际仓库在 E 盘
5. D 盘有 Windows 安全限制，git 操作可能遇到权限锁

八、后续修改注意事项
-------------------
- 修改任何版本时，只改需要改的那个版本，不要动另外两个
- 改完后 git add + commit + push，Render 自动部署
- 搜索功能的内存安全方案已验证可行，不要回退到 json.load() 全量加载
- LLM 版的前端 addMessage 函数中，先做 HTML 转义再做【标题】span 注入，顺序不能反
